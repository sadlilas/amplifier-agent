Positioning

One package, every consumer surface — that's the design goal

amplifier-agent is the official, versioned, supported distribution of Amplifier. The canonical way the kernel reaches a consumer — whether that consumer is a Node app, a desktop tool, an HTTP service, or a CI script.

One install path. One upgrade path. One trace boundary. Every host meets Amplifier through the same package.

Key point: The thesis of this talk is that amplifier-agent is not a product surface, not a demo, and not a SaaS. It is the canonical package — the official, versioned, supported way the Amplifier kernel reaches a consumer. Everything that follows in this deck flows from that framing: a package has consumers, a package has a contract, a package gets shipped.

If challenged: "Isn't this just an SDK?" Closer than 'SDK', but the framing matters — an SDK is what a consumer imports into their stack. amplifier-agent is the kernel itself, packaged and made portable. The wrappers (TS, Python) are the SDK surface; the binary is the package. The distinction becomes load-bearing two sections from now when we talk about upgrade and debugging — there is exactly one engine version in flight per host, ever.

Transition: Calling it "canonical" is a claim, not yet a result. The next slide asks: how do you actually validate that one package fits every consumer surface? Answer: not by theory.

Premise

Shipping anywhere requires knowing the shape — empirics, not theory

You cannot design a portable package's contract in a vacuum. The shape emerges from collision with real platforms, real constraints, and real consumers.

What "theory" would give us A clean wire diagram, plausible defaults, and a contract that survives until the first integration meeting.
What empirics give us The surface area that actually matters — the corners hosts push on, the assumptions they refuse to inherit, the abstractions that survive.

Key point: Portability is not a property you design in; it is a property you discover. Every host we integrate against reveals a constraint we did not previously know existed. The contract of amplifier-agent isn't what we wrote in the design doc — it is the subset of that design that survived contact with three real platforms.

If challenged: "Why not just write the spec and let integrators implement against it?" Because the spec is wrong on first contact — every time. We tried that with Mode B and deleted ~600 lines of streaming protocol when the first host couldn't use it. Empirics aren't a luxury; they're the only way to ship a package that doesn't fork on contact.

Transition: So the question becomes: which platforms? Not just any three — three that stress-test different parts of the contract. That's the next slide.

Confidence: High — this framing is consistent with the multi-host runway lock from the NC provider design doc and the Mode A pivot amendment. Empirical validation is the explicit project methodology.

The probes

Three targets, three platform classes — chosen to stress-test the shape

Probe I In-process · Node.js

nanoclaw

A TypeScript host that consumes amplifier-agent through the language wrapper, embedded inside its own runtime.

Stress-tests The wrapper API surface: can adapter authors program against spawnAgent → submit → AsyncIterable without leaking engine internals?
Probe II Local CLI · Subprocess

paperclip

A local tool that drives amplifier-agent as a child process — one invocation per turn, no long-lived state on the host side.

Stress-tests The raw wire: does argv in / JSON envelope out stay ergonomic when the host is the one writing the argv?
Probe III HTTP service · Response API

opencode

An HTTP service exposing a Response-API-compatible endpoint, with amplifier-agent sitting behind a network boundary.

Stress-tests The service-shape adaptation: can a stateless-subprocess engine sit behind a request/response API without leaking its model?

Key point: These three are not "three customers we happened to find." They were chosen deliberately as a basis set — three different integration shapes that, together, span the surface a portable package has to cover. Embedded library, subprocess CLI, networked service. If the contract survives all three, the contract is sound. If it survives only one or two, we know exactly which assumption is wrong.

If challenged: "Why these three and not, say, Claude Code or a hosted SaaS?" Coverage, not popularity. Claude Code is in the queue but is structurally similar to nanoclaw (in-process TS host). Hosted SaaS is structurally similar to opencode. The three we picked maximize *shape* coverage per unit of integration cost. Adding more after these three is incremental; before these three is speculative.

If challenged: "Are these three real integrations in flight, or design exercises?" Real. NC is on a locked feature branch with the AmplifierAgentProvider; paperclip and opencode are queued behind it. Status is the next section.

Transition: So those are the three probes. The natural next question is what each one has actually surfaced — what state each integration is in today. That's the next section.

Confidence: High on framing. The platform-class descriptions are derived from the wrappers in the repo (TypeScript + Python) and the documented integration patterns. The "stress-tests" column is editorial — it's the lens, not a quote.
